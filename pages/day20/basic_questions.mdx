# Basic Questions — Union Find

Six warm-ups. Trace the unions by hand before you implement them — DSU's whole win is muscle-memory speed.

<Steps>

### Question 1 — Trace naive union

Start with `parent = [0, 1, 2, 3, 4]` (5 singletons). Apply these unions in order using the **naive** template (no rank, no path compression, always `parent[rx] = ry`):

`union(0, 1)`, `union(2, 3)`, `union(0, 2)`, `union(0, 4)`.

What is the final `parent` array? What is `find(0)`?

<ShowHideSolution>
- `union(0, 1)`: `find(0)=0, find(1)=1`. `parent[0] = 1`. → `parent = [1, 1, 2, 3, 4]`.
- `union(2, 3)`: `find(2)=2, find(3)=3`. `parent[2] = 3`. → `parent = [1, 1, 3, 3, 4]`.
- `union(0, 2)`: `find(0)` walks `0 → 1` (root). `find(2)` walks `2 → 3` (root). `parent[1] = 3`. → `parent = [1, 3, 3, 3, 4]`.
- `union(0, 4)`: `find(0)` walks `0 → 1 → 3` (root). `find(4) = 4`. `parent[3] = 4`. → `parent = [1, 3, 3, 4, 4]`.

`find(0)` walks `0 → 1 → 3 → 4` → returns `4`.

**A chain of length 4** — exactly the kind of degradation that path compression solves.
</ShowHideSolution>

### Question 2 — Same trace, with path compression

Same start and same sequence, but use **path compression** in `find`. What's the final `parent` array, and what's `find(0)` after the last union?

<ShowHideSolution>
- `union(0, 1)`: `parent = [1, 1, 2, 3, 4]` (same as before — no path to compress yet).
- `union(2, 3)`: `parent = [1, 1, 3, 3, 4]`.
- `union(0, 2)`: `find(0) → 1` (root, no compression). `find(2) → 3`. `parent[1] = 3`. → `parent = [1, 3, 3, 3, 4]`.
- `union(0, 4)`: `find(0)` walks `0 → 1 → 3`. **Path compression**: `parent[0] = 3` and `parent[1] = 3`. → `parent = [3, 3, 3, 3, 4]`. `find(4) = 4`. `parent[3] = 4`. → `parent = [3, 3, 3, 4, 4]`.

After: `find(0)` reads `parent[0] = 3`, walks `3 → 4`. Path compression sets `parent[0] = 4`. Returns 4.

Now `parent = [4, 3, 3, 4, 4]`. (Element 1's parent is still 3, but next time we `find(1)` it'll get compressed too.)
</ShowHideSolution>

### Question 3 — Trace union by rank

5 singletons. Apply union by rank (without path compression):

`union(0, 1)`, `union(2, 3)`, `union(0, 2)`.

Show the parent and rank arrays after each step.

<ShowHideSolution>
Initial: `parent = [0, 1, 2, 3, 4]`, `rank = [0, 0, 0, 0, 0]`.

- `union(0, 1)`: roots are 0 and 1, both rank 0. Make 1 child of 0 (or vice versa — convention varies). `parent[1] = 0`, `rank[0] = 1`. → `parent = [0, 0, 2, 3, 4]`, `rank = [1, 0, 0, 0, 0]`.
- `union(2, 3)`: same shape. `parent[3] = 2`, `rank[2] = 1`. → `parent = [0, 0, 2, 2, 4]`, `rank = [1, 0, 1, 0, 0]`.
- `union(0, 2)`: roots are 0 (rank 1) and 2 (rank 1). Equal — pick either, increment the survivor. Say `parent[2] = 0`, `rank[0] = 2`. → `parent = [0, 0, 0, 2, 4]`, `rank = [2, 0, 1, 0, 0]`.

Note element 3 still points to 2, not directly to 0 — without path compression, the tree is `0 ← 2 ← 3`. A find from 3 will walk two steps.
</ShowHideSolution>

### Question 4 — Cycle detection

Given vertices `{0, 1, 2, 3}` and edges arriving in this order: `(0, 1), (1, 2), (2, 3), (3, 0)`. Use DSU to detect when a cycle is formed.

<ShowHideSolution>
- `(0, 1)`: `find(0) != find(1)`. Union. No cycle. `parent` now has them connected.
- `(1, 2)`: `find(1) != find(2)`. Union. No cycle.
- `(2, 3)`: `find(2) != find(3)`. Union. No cycle.
- `(3, 0)`: `find(3) == find(0)` (both in the same component now). **Cycle detected.**

The fourth edge would create a cycle, so we return `True` (or skip it, depending on the problem).
</ShowHideSolution>

### Question 5 — Counting components

A graph has 6 vertices `{0, 1, 2, 3, 4, 5}` and edges `(0, 1), (2, 3), (4, 5), (0, 2)`. How many connected components are there?

<ShowHideSolution>
Start: 6 components. Track the `components` counter, decrementing on every successful union.

- `(0, 1)`: union. components = 5.
- `(2, 3)`: union. components = 4.
- `(4, 5)`: union. components = 3.
- `(0, 2)`: roots are 0's root and 2's root — different. Union. components = 2.

Final: 2 components — `{0, 1, 2, 3}` and `{4, 5}`.
</ShowHideSolution>

### Question 6 — When NOT to use DSU

You're asked: "given a directed graph, return all vertices reachable from vertex `s`." Can DSU solve this?

<ShowHideSolution>
**No.** Two reasons:

1. DSU is **undirected** in spirit — it merges sets, with no notion of edge direction. A → B and B → A would both `union(A, B)`, even though they imply different reachability.
2. DSU answers "same group?" not "reachable from a specific source?" Two vertices in the same DSU component might or might not be reachable from `s` in a directed graph.

Use **DFS** or **BFS** instead. Mark `s` as visited, traverse out-neighbors, repeat.
</ShowHideSolution>

</Steps>
