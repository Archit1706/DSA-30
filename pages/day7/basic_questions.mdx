# Basic Heap Questions

Six warm-up exercises to lock in heap operations before you take on the interview problems. Each one tests a different muscle — the heap property, the index arithmetic, the build vs push tradeoff, the size-K trick.

If you can do these in your head, the practice problems will feel mechanical.

<Steps>

### 1. Is this a valid min-heap?

For an array to represent a valid **min-heap**, every parent at index `i` must be ≤ both `arr[2i+1]` (left child) and `arr[2i+2]` (right child) — for every index that has children.

Check `[3, 5, 9, 6, 8, 20, 10, 12, 18, 9]`:

<HeapVisualizer initialHeap={[3, 5, 9, 6, 8, 20, 10, 12, 18, 9]} kind="min" />

<ShowHideSolution>

Walk every index that has at least one child (`i < n/2`):

```text
i=0 (3): children at 1,2 → 5, 9 — both >= 3 ✓
i=1 (5): children at 3,4 → 6, 8 — both >= 5 ✓
i=2 (9): children at 5,6 → 20, 10 — both >= 9 ✓
i=3 (6): children at 7,8 → 12, 18 — both >= 6 ✓
i=4 (8): child at 9 → 9 — >= 8 ✓
```

Every parent is ≤ both children. **Valid min-heap.** ✅

Generic check in code:

```python copy
def is_min_heap(arr):
    n = len(arr)
    for i in range(n // 2):
        l, r = 2*i + 1, 2*i + 2
        if l < n and arr[i] > arr[l]: return False
        if r < n and arr[i] > arr[r]: return False
    return True
```

We only need to walk the **first half** of the array. Indices `n/2` through `n-1` are leaves — no children to compare against.

</ShowHideSolution>

### 2. Trace this sequence on an empty min-heap

Start with `[]`. Apply: `push(5), push(3), push(8), push(1), pop(), push(2), pop(), pop()`.

What does the heap look like at each step? What does each pop return?

<ShowHideSolution>

```text
push(5):  [5]
push(3):  [3, 5]            (3 sifts up past 5)
push(8):  [3, 5, 8]
push(1):  [1, 3, 8, 5]      (1 sifts up: swap with 5 → swap with 3)
pop():    [3, 5, 8]         returns 1; move last (5) to root → sift down
push(2):  [2, 3, 8, 5]      (2 sifts up past 3)
pop():    [3, 5, 8]         returns 2; move last (5) to root → sift down past 3
pop():    [5, 8]            returns 3
```

Three pops returned **1, 2, 3** — the three smallest values seen so far, in order. That's the entire heap promise.

</ShowHideSolution>

### 3. Build a max-heap from `[4, 10, 3, 5, 1]` — but two ways

Compare cost: building by **n successive pushes** vs **heapify in place**.

<ShowHideSolution>

**Method A — push each in turn:**

```text
push(4):  [4]                    cost: O(1)
push(10): [10, 4]                cost: O(log 1)
push(3):  [10, 4, 3]             cost: O(log 2)
push(5):  [10, 5, 3, 4]          cost: O(log 3)
push(1):  [10, 5, 3, 4, 1]       cost: O(log 4)
```

Total: O(n log n) — about 11 comparisons here.

**Method B — heapify in place:** start from `[4, 10, 3, 5, 1]`, sift down every node at index `< n/2` from right to left.

```text
i=1 (10): children at 3,4 → 5, 1 — already largest ✓
i=0 (4):  children at 1,2 → 10, 3 — swap 4 ↔ 10 → [10, 4, 3, 5, 1]
          then at i=1 (4): children 5, 1 — swap 4 ↔ 5 → [10, 5, 3, 4, 1]
```

Total: O(n) — about 4 comparisons here.

Both produce the same valid max-heap `[10, 5, 3, 4, 1]`. **Heapify wins** when you have all the elements up front. Push-by-push wins when elements arrive online.

</ShowHideSolution>

### 4. What's at index 5 in a complete binary tree of 11 nodes?

If the tree is stored at indices 0..10, where is the node at array index 5? Who is its parent? Children?

<ShowHideSolution>

- **Index 5** is the 6th node in level-order — depth 2 (since 5 ≥ 3 and 5 < 7).
- **Parent:** `(5 - 1) / 2 = 2` → index 2.
- **Left child:** `2*5 + 1 = 11` — out of bounds (n = 11). No left child.
- **Right child:** `2*5 + 2 = 12` — also out of bounds. No right child.

So index 5 is a **leaf** in this 11-node tree.

</ShowHideSolution>

### 5. K smallest using a heap — which kind?

You want the K smallest elements of a stream. Should you use a min-heap or a max-heap? What size?

<ShowHideSolution>

**Max-heap of size K.** The max-heap holds the K elements with the *smallest* values seen so far. The root is the **largest of the K-smallest** — i.e., the current threshold. A new element gets in iff it's smaller than the root; then evict the root.

```python copy
import heapq

def k_smallest(stream, k):
    heap = []                              # acts as max-heap via negation
    for x in stream:
        if len(heap) < k:
            heapq.heappush(heap, -x)
        elif -heap[0] > x:                 # current largest of K > new value
            heapq.heapreplace(heap, -x)
    return sorted(-v for v in heap)
```

The inverted intuition catches beginners: **"I want the smallest, why am I using a max-heap?"** Because the heap holds *winners* — the K-best we've seen so far — and the *loser among the winners* (the largest of the smallest K) is what gets evicted when a better candidate arrives.

The mirror image is true for K-largest: use a **min-heap of size K**, with the root as the smallest of the K-largest.

</ShowHideSolution>

### 6. After this max-heap pop, what does the heap look like?

Start with `[20, 15, 17, 8, 10, 5, 12]` (max-heap). Pop the root. Show the heap after the operation.

<HeapVisualizer initialHeap={[20, 15, 17, 8, 10, 5, 12]} kind="max" />

<ShowHideSolution>

Algorithm: save the top (20), move the **last element (12)** to the root, then **sift down**.

```text
After moving 12 to root: [12, 15, 17, 8, 10, 5]
                                ─ heap is broken: 12 < 15, 12 < 17

Sift down from index 0:
    children of 12: indices 1, 2 → values 15, 17
    largest child = 17 (index 2)
    swap 12 ↔ 17       →  [17, 15, 12, 8, 10, 5]

Sift down from index 2:
    children of 12: index 5 → value 5
    12 > 5, so stop.

Result: [17, 15, 12, 8, 10, 5]
```

Pop returned **20**, heap is now `[17, 15, 12, 8, 10, 5]`.

</ShowHideSolution>
